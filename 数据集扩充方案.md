## 一、 统一 JSON Schema 与字段字典

放弃传统的 CSV 格式，全面采用 `JSON Lines (.jsonl)`。新的 Schema 补充了溯源、置信度和高频字段顶层化设计。

### 1. 全局 JSON Schema 定义

```
{
  "$schema": "[http://json-schema.org/draft-07/schema#](http://json-schema.org/draft-07/schema#)",
  "title": "DeepSOC Unified Security Data Schema",
  "type": "object",
  "properties": {
    "_id": { "type": "string", "description": "全局唯一标识符，UUIDv4" },
    "db_type": { "type": "string", "enum": ["ioc", "cve", "web_attack", "strategy", "case"] },
    
    // 1. 核心业务字段 (允许缺失 / Nullable)
    "risk_level": { "type": ["string", "null"], "enum": ["Critical", "High", "Medium", "Low", "Info", null] },
    "mitre_attack_id": { "type": ["array", "null"], "items": { "type": "string" } },
    "tags": { "type": ["array", "null"], "items": { "type": "string" } },
    "search_content": { "type": "string", "description": "RAG 向量化专用的合成自然语言段落，必填" },

    // 2. 高频特有字段 (从 details 中上浮，便于直接建立索引或硬过滤)
    "cve_id": { "type": ["string", "null"] },
    "ioc_value": { "type": ["string", "null"] },
    "payload": { "type": ["string", "null"] },
    "affected_product": { "type": ["string", "null"] },

    // 3. 数据溯源与复现 (Traceability)
    "source": { "type": "string", "description": "顶层来源名称，如 NVD, ThreatFox" },
    "source_dataset": { "type": ["string", "null"], "description": "具体数据集名称或接口名称" },
    "source_url": { "type": ["string", "null"], "description": "原始数据的抓取链接" },
    "original_id": { "type": ["string", "null"], "description": "原始数据在源站的 ID" },
    "fetched_at": { "type": "string", "format": "date-time", "description": "抓取时间 ISO8601" },
    "raw_content_hash": { "type": "string", "description": "原始核心字段的 SHA256，用于精准去重" },
    "license": { "type": ["string", "null"], "description": "数据开源协议，如 CC-BY-4.0" },

    // 4. 置信度与标签来源 (Reliability)
    "confidence": { "type": ["integer", "null"], "minimum": 0, "maximum": 100 },
    "label_source": { "type": ["string", "null"], "enum": ["rule", "manual", "llm", "external", null], "description": "标签或评级的打标方式" },
    "verified": { "type": "boolean", "default": false, "description": "是否经过人工确认" },

    // 5. 长尾字段 (Details 边界)
    "details": {
      "type": ["object", "null"],
      "description": "仅限存放无法映射到上述顶层字段的长尾信息",
      "additionalProperties": true
    }
  },
  "required": ["_id", "db_type", "search_content", "source", "fetched_at", "raw_content_hash"]
}
```

### 2. 核心字段映射字典与规则

- **`risk_level` 映射规则**：
    
    - `Critical`: CVSS >= 9.0，或 ThreatFox 信心指数 >= 95，或已确认在野利用 (0-day)。
        
    - `High`: CVSS 7.0-8.9，或 ThreatFox 信心指数 75-94。
        
    - `Medium`: CVSS 4.0-6.9，或 ThreatFox 信心指数 50-74。
        
    - `Low`: CVSS 0.1-3.9，或常规扫描器探测。
        
    - `Info`: 资产信息、无害探测记录。
        
- **`mitre_attack_id` 提取规则**：
    
    - 使用正则 `T\d{4}(\.\d{3})?` 从原始情报描述中强制提取。
        
    - 利用外部映射表（如 NVD CVE 对应的 CWE 到 ATT&CK 的映射表）进行转换。
        
- **`tags` 规范与粒度**：
    
    - 禁止长句入标签。
        
    - 来源限于：恶意家族名（如 `Mirai`, `Lazarus`）、攻击向量（如 `SQLi`, `XSS`）、操作系统/组件名（如 `Linux`, `Nginx`）。
        
- **`details` 边界控制**：
    
    - **顶层字段定义**：用于检索、过滤、统计的稳定字段。
        
    - **细节字段定义**：`details` 只允许存放“本库独有且低频”的长尾字段。所有高频字段必须优先尝试映射到顶层。
        
    - **禁止**将 `cve_id`、`ioc_value`、`payload`、`affected_product` 等高频关键信息放入 `details`，它们必须存在于顶层。
        
- **`confidence` 与标签来源计算口径**：
    
    - **规则命中 (rule)**：强特征匹配，默认分配高置信度（如 90）。
        
    - **LLM 生成 (llm)**：由大模型推断或提取，默认分配中置信度（如 50-70）。
        
    - **外部源直接给定 (external)**：按源可信度折算（如 NVD 官方分配 95，普通开源社区库分配 60）。
        
    - **人工确认 (manual)**：当人工复核并将 `verified` 置为 `true` 后，置信度可上调至 100。
        

## 二、 分库最低可用校验规则与隔离区机制

不同的数据库有不同的生命线。在 ETL 的 Transform 阶段，如果不满足各自的最低字段要求，**不应直接全部丢弃（避免损失可用样本），而是引入隔离区（Quarantine）机制**：

- **正常入库**：满足最低字段集。
    
- **进入隔离区 (`quarantine/`)**：不满足最低字段集，但含有有效描述或部分参考特征，留存供后续人工抽查与数据增强。
    
- **直接丢弃**：核心内容完全为空的无效数据。
    

**各库正常入库的最低可用字段：**

1. **IOC 规则样本库** (`db_type: ioc`)
    
    - 必须存在：`ioc_value`（且格式必须合法，如有效 IP、域名或 MD5）。
        
2. **CVE & 漏洞情报库** (`db_type: cve`)
    
    - 必须存在：`cve_id`（必须符合 `CVE-\d{4}-\d+` 正则）。
        
3. **常见 Web 攻击模式库** (`db_type: web_attack`)
    
    - 必须存在：`payload` **或** `tags` 中包含具体的攻击类型模式。
        
4. **安全处置策略模板库** (`db_type: strategy`)
    
    - 必须存在：`details.title` **且** (`details.content` 或 `details.playbook_steps`)。
        
5. **安全案例库** (`db_type: case`)
    
    - 必须存在：`details.case_title` **或** `details.description`。
        

## 三、 数据去重与冲突合并机制(搜集数据时不用做，等所有人汇总之后再做)

为了防止重复样本污染 RAG 检索结果，在 Load (加载入库) 阶段必须执行去重合并流水线。

### 1. 完全重复 (Exact Match)

- **精确哈希定义 (`raw_content_hash`)**：
    
    - 计算公式：`SHA256(标准化后的 db_type + 关键字段 + source_dataset + original_id)`。
        
    - **标准化规范**：计算前必须将拼接串统一转小写、去除首尾空格、时间统一为 ISO8601 格式，并使用 UTF-8 编码，防止因格式差异导致哈希失效。
        
- **处理动作**：触发合并而非新增。更新 `fetched_at` 为最新时间，并追加 `source`（如原 source 为 `URLhaus`，新发现修改为 `URLhaus, ThreatFox`）。
    

### 2. 近似重复 (Fuzzy Match)

- **判断标准**：利用 ChromaDB 对新数据的 `search_content` 进行余弦相似度检索，若相似度 > 0.92（如同一 Web 攻击的不同变体 Payload，且描述高度一致）。
    
- **处理动作**：人工不介入时，作为独立记录保留，但在 `tags` 中自动打上 `variant` 或 `fuzzy_duplicate` 标签，并在检索召回时进行后处理去重。
    

### 3. 数据冲突处理 (Conflict Resolution)

- **场景**：同一个 IOC，A 源评定为 `High`，B 源评定为 `Low`。
    
- **处理动作**：依赖 `confidence` 和信源优先级配置（如 NVD > 商业威胁情报 > 开源社区项目）。
    
    - 主记录保留高优先级信源的值。
        
    - **必须将低优先级信源的原始判断完整保留**至 `details.conflict_history` 数组中，以备答辩溯源和模型综合研判。
        

## 四、 面向 RAG 的 `search_content` 模板

`search_content` 是向量库唯一读取的文本内容。为了防止向量检索被冗长的废话噪声稀释，必须统一采用**四段式（对象、发生、风险、建议）** 的精简检索摘要模板，长度严格控制在 80-200 字，**严格禁止将推测或空字段写进自然语言事实中**。

### 1. CVE 库模板

- **四段式结构**：【对象】+【发生】+【风险】+【建议】
    
- **示例**：
    
    > "【对象】Apache HTTP Server 2.4.49 组件。【发生】存在路径穿越漏洞(CVE-2021-41773)，攻击者可构造特定URL读取任意文件。【风险】评估为Critical(CVSS 10.0)，已确认存在在野利用。【建议】升级至 2.4.50 及以上版本。"
    

### 2. IOC 库模板

- **四段式结构**：【对象】+【发生】+【风险】+【建议】
    
- **示例**：
    
    > "【对象】恶意 URL 危害指标 `http://42.227.200.97:47576/bin.sh`。【发生】主要用于远控与木马分发意图，已知关联 Mirai 家族。【风险】当前置信度极高，风险等级为 High。【建议】在边界防火墙或 WAF 中下发阻断策略。"
    

### 3. Web 攻击模式库模板

- **四段式结构**：【对象】+【发生】+【风险】+【建议】
    
- **示例**：
    
    > "【对象】SQL注入攻击特征。【发生】典型 Payload 为正则 `updatexml\s*\(`，常在 HTTP GET 参数或 POST 表单中触发。【风险】可导致数据库信息泄露，风险 High。【建议】作为 WAF 规则配置与检测引擎的匹配基线。"
    

### 4. 处置策略与案例库模板

- **四段式结构**：【对象】+【发生】+【风险】+【建议】
    
- **示例 (策略)**：
    
    > "【对象】WebShell文件上传的处置策略。【发生】适用于 Nginx+WAF 架构场景。【风险】不及时处置将导致服务器权限彻底沦陷。【建议】核心执行步骤：确认目录执行权限，通过 SecRule 拦截 `\.(php|jsp)$` 写入，并隔离受感染主机。"
    

## 五、 外部 AI 助手在数据治理生命周期中的辅助应用指南

这里特指在**离线数据准备阶段**（非 DeepSOC 运行时后端），如何利用外部成熟的 AI 工具（如 Gemini、ChatGPT、豆包、Kimi 等）来极大加速数据的收集、代码编写和内容生成。

### 1. 数据搜集与爬虫脚本编写 (Discovery & Extraction)

不要手动去翻找 API 文档和写 Requests 爬虫。利用外部 AI 快速生成针对特定安全情报源的提取代码。

- **应用场景**：需要从 NVD、AlienVault OTX 或 GitHub 上的开源威胁情报仓库拉取数据。
    
- **AI 提示词模板 (Prompt)**：
    
    > "我需要构建一个安全情报库。请帮我写一段 Python 3 脚本，调用
    > 
    > $$ThreatFox / AlienVault OTX / NVD$$
    > 
    > 的公开 API。 目标：获取最近 7 天的
    > 
    > $$恶意 URL IOC / 高危 CVE$$
    > 
    > 。 要求：
    > 
    > 1. 使用 requests 库。
    >     
    > 2. 返回的数据不要直接打印，而是解析提炼出核心字段（如 ioc_value, threat_type, first_seen）。
    >     
    > 3. 代码要求极简、可直接运行，包含必要的注释解释逻辑。不需要任何 try-except 异常处理逻辑。"
    >     
    

### 2. 数据清洗与正则构建 (Cleaning & Alignment)

安全数据充斥着混杂的文本和非标准化的日志，利用 AI 编写正则表达式和 Pandas 处理逻辑是最高效的。

- **应用场景**：拿到一份脏乱差的 CSV，含有多余的引号、混合的中英文评级、嵌套的列表。
    
- **AI 提示词模板 (Prompt)**：
    
    > "我有一份不规范的 CSV 数据，包含以下列：
    > 
    > $$CVE编号, 危险程度, 影响版本$$
    > 
    > 。 '危险程度'一列目前很混乱，包含了：'严重', '高危', 'High', '10.0', '未知' 等值。 请帮我写一段 Python Pandas 代码：
    > 
    > 1. 将这些混乱的值统一映射为标准枚举值：
    >     
    >     $$'Critical', 'High', 'Medium', 'Low', 'Info'$$
    >     
    >     。
    >     
    > 2. 补齐缺失值，缺失统一填 null。
    >     
    > 3. 最终将清洗后的 DataFrame 导出为 JSON Lines (.jsonl) 格式。不要包含任何 try-except 块。"
    >     
    
- **生成高复杂度正则的 Prompt**：
    
    > "我需要提取一段无格式文本中的所有 MITRE ATT&CK 技术编号。编号的格式通常是 T 加上四位数字，可能带有子技术编号（如 T1548 或 T1548.002）。请给我一个极其严谨的 Python 正则表达式，并提供一段直接提取匹配项的简短代码。"
    

### 3. 半结构化数据解析与标签补全 (Tagging & Parsing)

对于《安全案例库》和《安全处置策略模板库》，原始资料往往是长篇大论的 PDF 或应急响应报告，直接入库毫无价值。

- **应用场景**：将长文本安全研判报告结构化。你可以把大段文本直接丢给 Kimi 或 ChatGPT，让它按照我们定义的 JSON Schema 吐出数据。
    
- **AI 提示词模板 (Prompt)**：
    
    > "你现在是一个专业的安全运营工程师。以下是一篇关于某次 APT 攻击的开源应急响应报告。 请阅读后，将其提炼为符合以下 JSON Schema 的结构： { "tags":
    > 
    > $$"列出不超过3个核心攻击向量或组织名"$$
    > 
    > , "mitre_attack_id":
    > 
    > $$"提取报告中提到的所有ATT&CK ID"$$
    > 
    > , "details": { "case_title": "...", "timeline":
    > 
    > $$"时间点1: 行为", "时间点2: 行为"$$
    > 
    > , "root_cause": "..." } } 请直接输出合法的 JSON 格式，不要包含任何 markdown 代码块标记和额外解释。"
    

### 4. 批量生成 RAG `search_content` (Narrative Generation)

如果你手头有几万条仅仅包含字段键值对的 JSON，你可以写一个简单的 Python 脚本，调用外部大模型的 API（或使用具备批量处理能力的 AI 工具），把冷冰冰的字段合成我们第四章要求的“标准叙事模板”。

- **应用场景**：根据字段拼接生成高质量的 `search_content`。
    
- **AI 提示词模板 (Prompt)**：
    
    > "我将提供一条结构化的 Web 攻击规则数据。请根据这些字段，合成一段简短的中文叙事，专供 RAG 向量检索使用。 【规则】：
    > 
    > 1. 结构必须遵循四段式：【对象】+【发生】+【风险】+【建议】。总字数控制在 80-150 字。
    >     
    > 2. 禁止编造输入数据中不存在的事实。
    >     
    > 3. 保持专业严肃的安全术语风格。 【输入数据】： db_type: web_attack payload: (?i)union._?select._?from attack_type: SQL注入 location: HTTP GET URL 【请输出合成的文本】："
    >
